KEY DESIGN DECISIONS
================================================================================

1. TWO-STAGE LLM PIPELINE
   - Stage 1: Extract criteria once per protocol (cached)
   - Stage 2: Evaluate each criterion individually
   - Rationale: Reduces API calls, improves caching, clearer logic

2. CONSERVATIVE INSUFFICIENT_DATA BIAS
   - Prefer "missing data" over assumptions
   - Prevents false negatives due to incomplete patient data
   - Requires clinician review and clarification
   - Ensures safety-first approach

3. NO DATA PERSISTENCE
   - Session state only (cleared on close)
   - No database of patient histories
   - Privacy-first design
   - Compliant with HIPAA concerns

4. SEPARATE FRONTEND & BACKEND
   - Frontend: React + Vercel (lightweight, fast)
   - Backend: FastAPI + Railway (scalable, serverless-friendly)
   - Decoupled: Easy to modify each independently
   - API-driven: Could support mobile apps in future

5. DE-IDENTIFICATION ENFORCEMENT
   - Validation blocks obvious identifiers
   - PHI warnings on all patient input screens
   - No persistent patient identifiers
   - Designed for research/clinical trial use

6. COMPREHENSIVE ERROR HANDLING
   - Specific HTTP status codes (400, 413, 422, 429, 500)
   - User-friendly error messages
   - Retry logic with exponential backoff
   - Graceful fallbacks (OCR when PDF parsing fails)
